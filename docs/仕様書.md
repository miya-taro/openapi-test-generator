## 1. 目的と適用範囲

### 1.1 目的

本書は、`openapi.json` を入力として「試験項目一覧 Excel」を自動生成するジェネレータの実装仕様を定義する。

ジェネレータは OpenAPI Specification 3.x の Validation Keywords のみを根拠とし、仕様に記載のない制約については一切のテストケースを生成しない。

### 1.2 適用範囲

- **生成対象**: `parameters`（`in: path / query / header / cookie`）および `requestBody`
- **生成根拠**: OpenAPI 3.x の Validation Keywords に限定
- **非対象**: ビジネスロジック、状態遷移、権限制御、DB制約 → 別レイヤ（YAML シナリオ）で管理
- **非対象**: `x-*` 拡張フィールド（カスタム拡張は本ジェネレータが無視する）

> 📌 **注意** 本ジェネレータが出力するテスト仕様は「Layer 1 インターフェース観点」のみです。Layer 2（業務観点）は YAML シナリオファイルで別管理してください。

## 2. アーキテクチャ概要

### 2.1 処理フロー

ジェネレータは以下の 5 フェーズで動作する。

| フェーズ | 処理名 | 入力 | 出力 | 備考 |
|---------|--------|------|------|------|
| 1 | Parse | `openapi.json` | OpenAPI IR（中間表現） | JSON Schema `$ref` を解決済みにする |
| 2 | Extract | OpenAPI IR | `ParamSpec[]` / `BodySpec[]` | Validation Keywords を抽出 |
| 3 | Generate | `ParamSpec[]` / `BodySpec[]` | `TestCase[]` | 本仕様書のルールに従い生成 |
| 4 | ID Assign | `TestCase[]` | `TestCase[]`（ID付き） | `TC-{operationId}-{seq3桁}` |
| 5 | Render | `TestCase[]` | `test_spec.xlsx` | Excel カラムにマッピング |

### 2.2 主要データ構造

#### ParamSpec（パラメータ仕様）

```typescript
interface ParamSpec {
  operationId:  string;
  path:         string;
  method:       string;
  in:           'path' | 'query' | 'header' | 'cookie';
  name:         string;
  required:     boolean;
  schema:       ResolvedSchema;  // $ref 解決済み
}
```

#### BodyFieldSpec（リクエストボディフィールド仕様）

```typescript
interface BodyFieldSpec {
  operationId:  string;
  path:         string;
  method:       string;
  fieldPath:    string;   // 例: 'user.address.zipCode'
  required:     boolean;
  schema:       ResolvedSchema;
}
```

#### TestCase（生成済みテストケース）

```typescript
interface TestCase {
  id:             string;   // TC-getUser-001
  operationId:    string;
  summary:        string;
  path:           string;
  method:         string;
  in:             string;
  paramName:      string;
  keyword:        string;   // 根拠 Keyword
  perspective:    string;   // 観点ラベル
  inputValue:     string;
  expectedStatus: string;
  expectedResult: string;
  verdict:        '';       // 判定は空（手動記入）
  notes:          string;
}
```

## 3. ケース生成ルール（Validation Keyword 別）

> 📌 以下のルールに登場しない Keyword（例: `description`, `example`, `default`）は生成対象外。ジェネレータは該当フィールドに対してテスト行を出力しない。

### 3.1 共通ルール（全型・全 in）

| Keyword | 観点 | 入力値 | 期待ステータス | 期待結果 | ケース数 |
|---------|------|--------|----------------|----------|---------|
| `required: true` | 必須フィールド欠落（異常） | フィールドを省略 | 400 | バリデーションエラーが返ること | 1 |
| `type` | 型不正値（異常） | integer → `"abc"`、boolean → `"yes"` | 400 ※1 | 型エラーが返ること | 1 |

※1 `path` パラメータの型不正は、ルーティング失敗により 404 になる場合がある（実装依存）。備考欄に明記すること。

### 3.2 integer / number 型

| Keyword | 観点 | 入力値の算出式 | 期待ステータス | 備考 |
|---------|------|----------------|----------------|------|
| `minimum` | 最小値（正常） | `minimum` | 200 | |
| `minimum` | 最小値未満（異常） | `minimum − 1` | 400 | integer は整数減算、number は小数点以下を考慮 |
| `maximum` | 最大値（正常） | `maximum` | 200 | |
| `maximum` | 最大値超過（異常） | `maximum + 1` | 400 | |
| `minimum` + `maximum` | 中間値（正常） | `⌊(minimum+maximum)÷2⌋` | 200 | 両方が存在する場合のみ追加 |
| `exclusiveMinimum`（数値） | 境界値そのもの（異常） | `exclusiveMinimum` | 400 | OAS 3.1: `exclusiveMinimum` が数値 |
| `exclusiveMinimum`（数値） | 境界値+1（正常） | `exclusiveMinimum + 1` | 200 | |
| `exclusiveMaximum`（数値） | 境界値そのもの（異常） | `exclusiveMaximum` | 400 | |
| `exclusiveMaximum`（数値） | 境界値−1（正常） | `exclusiveMaximum − 1` | 200 | |
| `exclusiveMinimum`（boolean） | minimum そのもの（異常） | `minimum` | 400 | OAS 3.0: `minimum` + `exclusiveMinimum: true` |

> 📌 `multipleOf` が宣言されている場合は、`multipleOf` の倍数（正常）と非倍数（異常）をそれぞれ 1 件追加する。

### 3.3 string 型

| Keyword | 観点 | 入力値の算出式 | 期待ステータス | 備考 |
|---------|------|----------------|----------------|------|
| `enum` | enum 代表値（正常） | `enum[0]` | 200 | 全 enum 値を網羅したい場合は cases を `enum.length` に |
| `enum` | enum 外の値（異常） | `"__INVALID__"` | 400 | |
| `maxLength` | 最大長（正常） | `"a".repeat(maxLength)` | 200 | |
| `maxLength` | 最大長+1（異常） | `"a".repeat(maxLength + 1)` | 400 | |
| `minLength` | 最小長（正常） | `"a".repeat(minLength)` | 200 | |
| `minLength` | 最小長−1（異常） | `minLength > 1 ? "a".repeat(minLength - 1) : ""` | 400 | `minLength = 0` の場合は生成不要 |
| `pattern` | パターン一致（正常） | 正規表現から逆生成（※2） | 200 | 逆生成不可の場合は手動入力欄を設ける |
| `pattern` | パターン不一致（異常） | `"__BAD_PATTERN__"` | 400 | |
| `format: date` | 適合値（正常） | `"2024-01-01"` | 200 | format 値ごとに固定サンプルを定義（付録 A 参照） |
| `format: *` | 不適合値（異常） | `"not-a-date"` | 400 | サーバが format をバリデートしない場合 200 の可能性あり |

※2 `pattern` の逆生成: [RandExp](https://github.com/fent/randexp.js) などのライブラリを用いる。逆生成が失敗する場合（バックトラッキング系の複雑な正規表現等）は `inputValue` を `"[手動入力]"` とし、Excel の備考欄に正規表現を記載する。

### 3.4 array 型

| Keyword | 観点 | 入力値の算出式 | 期待ステータス | 備考 |
|---------|------|----------------|----------------|------|
| `minItems` | 最小要素数（正常） | 要素数 = `minItems` の配列 | 200 | |
| `minItems` | 最小要素数−1（異常） | 要素数 = `minItems − 1` の配列 | 400 | |
| `maxItems` | 最大要素数（正常） | 要素数 = `maxItems` の配列 | 200 | |
| `maxItems` | 最大要素数+1（異常） | 要素数 = `maxItems + 1` の配列 | 400 | |
| `uniqueItems: true` | 重複要素あり（異常） | `["a", "a"]` | 400 | `items` の `type` に合わせた重複値を使用 |

### 3.5 object 型（requestBody ネストオブジェクト）

ネストしたオブジェクトの各プロパティに対して、フラットな `ParamSpec` と同じルールを再帰的に適用する。

| Keyword | 観点 | 入力値 | 期待ステータス | 備考 |
|---------|------|--------|----------------|------|
| `required`（プロパティ） | 必須プロパティ欠落（異常） | 該当プロパティを省略した JSON | 400 | |
| `additionalProperties: false` | 未定義プロパティ追加（異常） | スキーマ外のキーを含む JSON | 400 | `additionalProperties` が `false` の場合のみ |
| （空 body） | 空 body / null body（異常） | `{}` または `null` | 400 | `required` プロパティが 1 件以上ある場合のみ生成 |

## 4. in 別の固有処理

### 4.1 path パラメータ

- OpenAPI 仕様により `required: true` が必須。`required` の明示がなくても必須として扱う。
- **欠落テスト**: パラメータを省略すると別 path にルーティングされるため期待ステータスは `404` を設定し、備考に「ルーティング失敗により 404 の可能性あり」と記載する。
- **型不正テスト**: 期待ステータスは `400` を基本とし、備考に「ルーティング失敗により 404 になる場合あり（実装依存）」を付記する。
- URL エンコードが必要な文字（スペース、スラッシュ等）を含む `enum` 値は `inputValue` をエンコード済みの値とし、備考に元の値を記載する。

### 4.2 query パラメータ

- `required: false` の場合: パラメータを省略するケース（正常系）を 1 件追加し、観点を「省略時デフォルト動作確認」とする。
- `style` / `explode` が明示されている場合: フォーマット依存の入力値を使用する（例: `form` + `explode=true` → `key=a&key=b`）。
- `style` / `explode` が未指定の場合: OpenAPI 3.x のデフォルト（`form` + `explode=true`）を適用する。

### 4.3 header パラメータ

- `Authorization`、`Content-Type`、`Accept` などの標準 HTTP ヘッダは本ジェネレータの対象外とする（セキュリティ・コンテンツネゴシエーション仕様で別管理）。
- HTTP ヘッダ名の大文字小文字は区別しない（RFC 7230）。テストの入力値に影響しないが、備考欄に記載する。
- `required: true` のカスタムヘッダを省略した場合の期待ステータスは `400` または `401` とし、API 仕様の記述に従う。

### 4.4 cookie パラメータ

- セッション ID・CSRF トークンなど動的に生成される Cookie 値は本ジェネレータの対象外。YAML シナリオで前提条件として設定する。
- `Secure` / `HttpOnly` / `SameSite` などの Cookie 属性はインターフェース観点外。セキュリティテスト仕様で別管理。
- Cookie の型・長さ・`enum` 値のバリデーションのみを本ジェネレータが扱う。

### 4.5 requestBody

- Content-Type が `application/json` のみを対象とする。`multipart/form-data`、`application/x-www-form-urlencoded` は本ジェネレータ対象外。
- ネストしたオブジェクトのプロパティは `fieldPath`（例: `user.address.zipCode`）で表現し、再帰的にケースを生成する。
- `type: string, format: binary` のフィールド（ファイルアップロード）は対象外とし、備考に「ファイルアップロードテスト仕様で別管理」と記載する。
- `required` な `requestBody` 全体が欠落した場合（`Content-Length: 0`）の異常系を 1 件追加する。

## 5. 試験 ID の採番規則

試験 ID は以下のフォーマットで採番する。

```
TC-{operationId}-{seq}
```

| 構成要素 | 説明 | 例 |
|---------|------|----|
| `TC` | 固定プレフィックス | `TC` |
| `operationId` | OpenAPI の `operationId` をそのまま使用（キャメルケース可） | `getUser` |
| `seq` | `operationId` ごとに `001` からの 3 桁連番 | `001`, `002`, ... |

- `operationId` が未設定の場合は `{METHOD}_{PATH_NORMALIZED}` を使用する（例: `GET /users/{id}` → `GET_users_id`）。
- `PATH_NORMALIZED`: パスパラメータの `{}` を除去し、`/` を `_` に置換する。
- `seq` はパラメータ名・Keyword・正常/異常の順でソートした後に採番することで、再生成時の ID の安定性を確保する。

## 6. エラーハンドリングと警告

### 6.1 入力検証

| 条件 | ジェネレータの動作 | ログレベル |
|------|-------------------|-----------|
| `openapi.json` が OpenAPI 3.x 以外 | 処理中断・エラー終了 | `ERROR` |
| `$ref` が解決できない | 該当 operation をスキップ・処理継続 | `WARN` |
| `operationId` が重複 | 後続の operation に `_2`, `_3` サフィックスを付与 | `WARN` |
| `operationId` が未設定 | `PATH_NORMALIZED` ルールで代替 | `INFO` |
| `pattern` の逆生成失敗 | `inputValue` を `"[手動入力]"` とし処理継続 | `INFO` |

### 6.2 スキップ対象

- `type: string, format: binary`（バイナリ）フィールド
- `x-*` 拡張フィールド
- Content-Type が `application/json` 以外の `requestBody`
- `minLength: 0` の `minLength-1` テスト（長さ -1 は不正のためスキップ）

## 7. Excel 出力仕様

### 7.1 カラム定義

| 列名 | データ型 | 生成元 | 備考 |
|------|---------|--------|------|
| 試験ID | string | 採番ルール（5章） | `TC-{operationId}-{seq}` |
| operationId | string | `openapi.json` の `operationId` | 未設定時は `PATH_NORMALIZED` |
| 概要 | string | ジェネレータが自動生成 | 例: `"name 最大長超過（異常）"` |
| パス | string | path テンプレート | 例: `/users/{id}` |
| メソッド | string | HTTP メソッド | 大文字 |
| in | string | `parameter` の `in` フィールド | `path` / `query` / `header` / `cookie` / `body` |
| パラメータ名 | string | `parameter.name` / `fieldPath` | ネストは `"user.address.zipCode"` 形式 |
| Keyword | string | 根拠 Keyword | 例: `maxLength`, `enum`, `minimum` |
| 観点 | string | 観点ラベル | 例: `正常系_最大長` / `異常系_最小値未満` |
| 入力/操作 | string | 算出ロジック（3章） | 具体的な入力値文字列 |
| 期待ステータス | string | 3章のルール | `200` / `400` / `404` など |
| 期待結果 | string | 3章のルール | 日本語の期待動作説明 |
| 判定 | string | 空白（手動記入） | `OK` / `NG` / `未実施` |
| 備考 | string | ジェネレータが必要に応じて付記 | 実装依存注記など |

### 7.2 シート構成

- シート名: `test_spec`（固定）
- 1行目: ドキュメント情報（ファイル名・生成日時・`openapi.json` のバージョン）
- 2行目: 注意書き（インターフェース観点のみ、Layer 2 は YAML シナリオで管理する旨）
- 3行目: カラムヘッダ（フリーズ）
- 4行目以降: テストケース（`operationId` → `in` → パラメータ名 → 正常/異常 の順でソート）

### 7.3 書式

- 正常系行: 背景色 `#E2EFDA`（薄緑）
- 異常系行: 背景色 `#FCE4D6`（薄オレンジ）
- カラムヘッダ行: 背景色 `#1F3864`（濃紺）、文字色 白、太字
- フォント: Arial 10pt
- 「判定」列はドロップダウン（`OK` / `NG` / `未実施`）を設定する

## 8. CLI インターフェース

| コード | 意味 |

## 付録 C: レイヤ分離の責務境界

> 📌 本ジェネレータが出力するテスト仕様（Layer 1）は、API のインターフェース契約（型・制約・必須性）のみを検証します。業務シナリオ・権限・状態遷移は Layer 2（YAML シナリオ）で管理してください。

| 観点 | Layer 1（本ジェネレータ） | Layer 2（YAML シナリオ） |
|------|--------------------------|------------------------|
